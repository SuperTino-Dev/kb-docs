- Slow Distribution Path performance

- WAN instability
- MTU fragmentation
- TCP retransmissions
- Receiver bottlenecks
- Disk I/O issues
- Apply-side slowness
- VPN/WireGuard issues

This version of the document includes additional validation steps and observations based on real troubleshooting results.

## Section 1 — Basic Network Connectivity Validation

### Purpose

The following commands validate whether the source server can reach the target server over the network.

This helps identify:

- packet loss
- routing issues
- unstable WAN links
- latency problems

---

### Validate Basic Connectivity

#### Command Explanation

The `ping` command sends ICMP packets to the target server and measures:

- latency
- packet loss
- response consistency

---

#### Command

```bash
ping target-ip
```

---

#### Expected Results

```text
Healthy Results
----------------------------------------
0% packet loss
Stable latency
Consistent response times
```

---

#### Analysis

Typical GG WAN latency guidance:

```text
Latency              Expectation
-------------------  ------------------------
0-10 ms              Excellent
10-40 ms             Very Good
40-100 ms            Good
100-200 ms           Acceptable
200+ ms              Potential Issues
```

WAN latency below 100 ms is generally considered acceptable for GoldenGate replication.

---

### Additional Validation

## Section 2 — Network Path Validation

### Purpose

The following commands validate:

- network path stability
- MTU consistency
- routing behavior
- intermediate network hops

---

### Validate Network Path

#### Command Explanation

The `tracepath` command displays the network path between source and target while also validating MTU discovery.

---

#### Command

```bash
tracepath target-ip
```

---

#### Expected Results

```text
Resume: pmtu 1500
```

---

#### Analysis

This confirms:

```text
Path MTU = 1500
```

which is considered healthy.

The multiple:

```text
no reply
```

entries are typically caused by routers that block ICMP responses and do NOT necessarily indicate a problem.

---

#### Important Observation

The final message:

```text
Too many hops: pmtu 1500
```

does NOT indicate MTU failure.

It simply means the destination stopped replying to tracepath probes.

---

### Additional Validation

#### Command Explanation

The following command validates whether packets of 1500 bytes can traverse the network WITHOUT fragmentation.

This is critical for:

- VPN replication
- WireGuard
- WAN replication
- GoldenGate throughput

---

#### Command

```bash
ping -M do -s 1472 target-ip
```

---

#### Expected Results

```text
1472 payload successful
No fragmentation
Stable latency
```

---

#### Analysis

This confirms:

```text
MTU 1500 is fully functional
```

No fragmentation issues detected.

This significantly reduces the likelihood that the GG Pump problem is caused by:

- MTU mismatch
- VPN fragmentation
- packet fragmentation

## Section 5 — TCP Health Validation

### Purpose

Validate TCP socket stability and identify reconnect storms or TCP churn.

---

### Validate TCP Socket Summary

#### Command Explanation

The `ss -s` command displays a summary of TCP socket states.

This helps identify:

- reconnect storms
- excessive TIME_WAIT sockets
- TCP instability
- socket exhaustion

---

#### Command

```bash
ss -s
```

---

#### Expected Healthy Results

```text
Low TCP counts
Low TIME_WAIT
No orphaned sockets
Stable ESTAB connections
```

---

#### Analysis

Healthy output generally indicates:

- stable WAN connectivity
- low TCP churn
- stable replication sessions

---

#### Potential Warning Signs

```text
Very high TIME_WAIT
Large ESTAB counts
Many orphaned sockets
```

These may indicate:

- reconnect storms
- unstable WAN
- VPN instability
- firewall resets
- application instability

---

## Section 6 — Additional TCP Session Validation

### Purpose

Validate active TCP sessions related to GoldenGate.

---

### Validate Active Sessions

#### Command Explanation

The following command displays active TCP sessions and queue sizes.

This helps identify:

- stalled send queues
- blocked receivers
- TCP congestion
- receiver bottlenecks

---

#### Command

```bash
ss -tanp
```

---

#### What To Look For

##### Healthy

```text
ESTAB
Send-Q = low
Recv-Q = low
```

---

##### Potential Problems

```text
SYN-SENT
```

Possible causes:

- firewall
- routing issue
- target unreachable

---

##### Another Important Indicator

```text
High Send-Q
```

This may indicate:

```text
sender transmitting faster than receiver can consume
```

Possible causes:

- slow receiver
- WAN congestion
- target disk bottleneck
- apply-side slowness
